Page Description
Example 1 To begin with I have simply consumed the JSON on the server. The page is then rendered on the server before it is sent to the user.
Example 2 Then to prevent excessive calls to the API I have made use of a cache. In this example API calls are cached for up to 5 minutes by Play. I have also added show and hide toggle links to the page.
Example 3 This final example uses AJAX to load the bill into the page. A few extra UI effects have been added and effort has been made to make the page presentable on mobile devices.
JS Tests During this project I have learned the basics of the QUnit Testing Framework. This page contains a test runner that performs tests on the javascript used in Example 3.
Github Source code is hosted on GitHub.

Assumptions

  • The API hosted on Heroku at http://safe-plains-5453.herokuapp.com/bill.json will always be available.
  • The JSON received by the application will always contain the same structure as the example Json provided for this project. For example, if a bill contains no telephone calls, the calls array property will still be present, just empty.
  • For the first two examples, the number of subscriptions, calls, rentals and buyAndKeeps will always > 0.
  • Ajax calls never fail. (If only...)
  • Latest version of Firefox or Chrome will be used to view the bill on a desktop or mobile device. (Again, if only...)

Possible further steps

  • Implement client and server side failure handling. What if the JSON returned from the API is erroneous?
  • Improve accessibility.
  • Support for browsers with disabled Javascript.
  • Graphics, favicon etc.
  • Custom error pages.
  • Improve support for older browsers.
  • Use websockets to stream live bill changes to the user in realtime.
  • Learn Angular and reimplement the front end!
